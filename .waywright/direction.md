# Direction Decision

## Goal
Help developers understand and trust autonomous code changes in this repository.

## Recalled Engineering Memory
- **D-000: Shared engineering memory with a cognitive-debt UI** — The agent and humans should use the same record of decisions, rejected alternatives, and real execution outcomes. The agent uses it to choose what to build next; humans use it to reconstruct why.
  - Outcome: Selected for implementation during the hackathon.
  - Lessons: A remembered decision must include why, what was rejected, and what actually happened.; Human understanding is one output edge; memory-grounded direction finding remains the core behavior.

## Candidates
### Render decision-linked diff annotations in PR/commit view **SELECTED**
Surfacing each hunk's originating decision (why, what was rejected, what happened) inline in the diff reduces reviewer time-to-trust more than a plain PR summary

Acceptance: Diff view renders a rationale marker per changed hunk pointing to its memory record; Marker shows selected direction + at least one rejected alternative + outcome status; Manual test: reviewer can answer 'why was X changed' without leaving the diff for 3 sample PRs

### CLI/script to query 'why' for any file or line from stored decision records
A queryable command (e.g. `why <file>:<line>`) grounded in typed decision records is more useful for trust-building than passive documentation since it works on-demand during code review, not just at merge time

Acceptance: Command resolves a file/line or commit SHA to its decision record(s) via git blame + memory lookup; Output includes rationale, rejected alternatives, and evidence fields verbatim (no LLM restatement, per D-000/4151); Works for at least the last 5 real commits in this repo as a smoke test

### Human-readable explanation of merge-gate pass/fail tied to Pomerium policy decisions
Since merge safety already relies on a fail-closed Pomerium boundary (per memory), exposing *why* a specific autonomous change passed or was blocked builds trust at the highest-stakes moment (merge time) more directly than general diff annotation

Acceptance: Each merge attempt logs which policy rule allowed/blocked it plus the decision record that motivated that policy; Blocked merges show a human-readable reason distinct from raw Pomerium/HTTP status; Verified against at least one real allowed and one real blocked merge in this repo's history

## Selected: Render decision-linked diff annotations in PR/commit view
D-000 established that shared engineering memory (decisions + rejections + outcomes) is the differentiator, and its explicit consumer edge is 'humans use it to reconstruct why.' annotated-diff-rationale is the direct implementation of that human-facing edge: it surfaces the decision record at the exact point (diff/PR view) where a developer is trying to trust an autonomous change. Sourcify's prior art gives a concrete, already-proven template — typed status enum + structured transformation list + closed-set error codes + decoupled result-object-from-presentation — for exactly the payload this direction needs per hunk, and their MIT-licensed, actively maintained repo (941 stars, TS-heavy) is a legitimate pattern source. This is a natural evolution of D-000's rationale, not a reversal.

## Rejected
- decision-record-cli: An on-demand CLI is a weaker default UX for building trust than diff annotations: it requires the developer to already suspect something and issue a manual query, whereas the goal is fluent, unavoidable exposure of rationale during code review. No prior-art evidence was supplied for this candidate, unlike the other two, indicating shallower validation.
- merge-gate-explainer: Targets a narrower, already-covered edge (per memory 4148, the Pomerium fail-closed boundary is established) — it explains policy pass/fail, not general code trust. It's a good future direction but is downstream of and dependent on decision records already being rendered legibly, which is exactly what annotated-diff-rationale builds first. Its prior art (OPA signed traces) is also heavier infrastructure (pluggable Signer/JWT) than the hackathon scope likely affords.

## Evidence
- memory D-000: 'humans use it to reconstruct why' names diff/PR-time explanation as the human consumer edge of shared engineering memory
- priorArt ethereum/sourcify: transformations/transformationValues list is a ready template for per-hunk 'what was normalized/rejected and why' payloads
- priorArt ethereum/sourcify: closed VerificationErrorCode enum enables consistent, filterable 'rejected: X' badges instead of free-text parsing
- priorArt ethereum/sourcify: lib-sourcify/server decoupling shows structured decision objects can feed multiple views (API, PR annotation, DB) without re-deriving rationale at render time
- context repository: demo-target has no existing scripts/pipeline, so a diff-rendering feature is greenfield and low-conflict to add now
